## Scope

Three independent bugs reported against 1.4.5. Each is fixed in isolation; nothing in this spec depends on rolling out the others.

| # | Symptom | Root cause | Fix |
|---|---------|------------|-----|
| 1 | After re-enabling a disabled grouped rule from the panel, the asset row vanishes from the panel after a page refresh. | Panel's asset list comes from `$wp_scripts->queue` at `wp_enqueue_scripts:999` plus a fallback that only includes a row when an active rule still matches. After the rule is deleted, an asset enqueued at a higher priority or via `wp_footer` is no longer covered by the fallback, so the row disappears. | Include `cu_group_items` snapshots from **enabled** groups whose URL pattern matches the current URL into the panel's `assets[]`, so user-managed assets always have a row regardless of queue-scan timing. |
| 2 | After AI Assets Scanner pushes rules, an open CU admin Rules tab does not refresh — total count and table stay stale until manual reload or tab switch. | Scanner writes rules server-side via `RuleRepository::create_rule()`; nothing emits a browser-side event. CU admin already listens for `cu.rule.changed` on `BroadcastChannel('code-unloader')` (via `cu-bus.js`), but no producer fires it on Scanner pushes. | Have Scanner's push-success JS post a `cu.rule.changed` message on `BroadcastChannel('code-unloader')`. The existing CU listener handles the rest. |
| 3 | The CU panel shows a different filename than the browser's network tab for what looks like the same asset (e.g. panel shows `animo.min.js`, network shows `eb-animation-load.js`). | Panel's filename is derived from `$wp_scripts->registered[$handle]->src` — the developer-registered URL. The browser sees the URL after `script_loader_src` / `script_loader_tag` filters run (e.g. Perfmatters bundling, hash-versioning). They can be the same script under a rewritten URL, OR two different scripts both matching the user's filter string. | Hook `script_loader_src` and `style_loader_src` at `PHP_INT_MAX` to capture the post-filter URL. When it differs from the registered src, surface it in the panel row as a second line ("→ rewritten: …"). Always show the original handle and registered filename so behaviour stays predictable. |

## Bug 1 — Show snapshot-backed assets in the panel

### Goal

Once a user has captured an asset into a CU group (via the panel "Save Rule" flow or via Scanner push), the asset row stays visible on every URL where the group's snapshot URL pattern matches — regardless of whether an active rule currently exists for it. Toggling Disabled → Active no longer makes the row disappear after a refresh.

### Behaviour change

For each URL that the panel renders on:

1. The current logic (already in 1.4.5) collects assets from `$wp_scripts->queue` + `$wp_styles->queue`, then adds a `was_dequeued: true` row for each `rule_map` entry whose handle is registered but absent from the queue.
2. **New step:** also collect `cu_group_items` snapshots whose `group_id` belongs to a group with `enabled = 1` AND whose snapshot URL pattern matches the current URL via `PatternMatcher::match()`. For each such snapshot whose `(handle, asset_type)` is not already present in `assets[]`, add a row with:
   - `handle`, `asset_type` from the snapshot row.
   - `source_label`: prefer `AssetDetector::detect_source($obj->src)` when `$obj = $wp_scripts->registered[$handle]` / `$wp_styles->registered[$handle]` exists; fall back to the snapshot's stored `source_label` column when the handle isn't registered on this URL. This keeps deactivated-plugin snapshots from collapsing into "Unknown / External" if the original push recorded a useful label.
   - `size`, `src`, `deps` populated from the registered object when available; `0` / `''` / `[]` otherwise.

   When `(handle, asset_type)` is **already** in `assets[]` (live queue or `was_dequeued` fallback), do not add a duplicate — the existing row is authoritative.

   No `from_snapshot: true` flag is added in 1.4.6 — it had no consumer and would be dead data. If a future release needs to differentiate snapshot-only rows in the UI, the flag is added in that release alongside the consumer.

3. The badge logic in `panel.js` is unchanged: a row with no `rule_map` entry shows green/Active. A snapshot-backed row with no active rule therefore renders as Active — which is correct, because no rule is dequeueing it on this URL.

### Why "enabled groups only"

A disabled group's snapshots are intentionally inert (the user has switched the group off). Showing inert snapshots as panel rows would surface assets the user has explicitly hidden from the active set. Disabled-group rows would also conflict with the existing rule_map filter (line 168 of `FrontendPanel.php`) which already skips disabled-group rules.

### Out of scope

- Surfacing snapshots as a distinct UI section ("Tracked but inactive"). 1.4.6 keeps the existing single list; the row simply appears as Active when no rule is active.
- Pruning orphan snapshots (snapshots whose handle is no longer registered in WordPress). Already handled separately by `get_stale_rule_ids()` for active rules; not extended to snapshots here.
- **Device-type and condition filtering of snapshot rows.** Snapshots stored with `device_type='mobile'` or a `condition_type` are merged into `assets[]` regardless of the panel viewer's device or whether the condition currently evaluates true. This matches today's `rule_map` behavior (the existing fallback at `FrontendPanel.php:178-205` does not condition-filter either) and avoids the user surprise where a row hides itself based on conditions the user can't see. If a future release wants device/condition-aware row visibility, it should change both the snapshot merge and the existing `rule_map` consistently.

### Edge cases

- **Snapshot handle is not registered in WordPress on this URL.** The row is still added (matches today's `was_dequeued` behavior of skipping when `!$obj`, except now we don't skip — we use empty `src`/`deps` so the row shows with handle and type only). Trade-off: a snapshot for a plugin that's been deactivated would still show a row. Acceptable — it tells the user "you have a snapshot for X but the plugin isn't installed/active here." If this proves noisy in practice, a follow-up can add a "stale" badge.
- **Same handle has both JS and CSS snapshots in different groups.** Already keyed on `handle|type`, so both rows appear independently — same as today.
- **Snapshot exists in two enabled groups for the same URL.** Deterministic resolution: the new query orders by `group_id ASC, id ASC`, so the snapshot in the lowest-numbered (oldest) group wins. Reproducibility matters more than fairness here — the user can deduplicate by managing groups; the panel does not try to be smart.

### Files

- `src/Core/RuleRepository.php` — new public method `get_group_item_snapshots_for_url(string $url): array` returning rows from an `INNER JOIN` of `cu_group_items` × `cu_groups` on `g.id = gi.group_id AND g.enabled = 1`, ordered by `gi.group_id ASC, gi.id ASC` for deterministic two-group tie-breaking, then filtered in PHP by `PatternMatcher::match()` (mirrors how `DequeueEngine::process_rules` filters). Object-cached per request under key `cdunloader_snapshots_for_url_<md5>`.
- `src/Frontend/FrontendPanel.php` — call the new method inside `enqueue_panel_assets()` after the `was_dequeued` fallback loop; merge into `$assets[]`.

## Bug 2 — Scanner push notifies open CU admin Rules tab

### Goal

When the AI Assets Scanner finishes a successful rule push to CU, any open CU admin Rules tab (same browser, same origin) refreshes its list table and total count in place — same UX the panel already gets via `cu-bus.js`.

### Behaviour change

In the Scanner plugin's admin JS (the script that fires the push REST call), on success of the push response, emit:

```js
try {
    var bc = new BroadcastChannel('code-unloader');
    bc.postMessage({ type: 'cu.rule.changed', source: 'scanner', action: 'bulk-create' });
    bc.close();
} catch (e) { /* BroadcastChannel unsupported — skip */ }
```

The CU admin listener (`assets/js/admin.js`'s `wireCrossTabSync`) already:

- Listens for `cu.rule.changed` on the bus.
- Ignores its own echoes (`msg.source === 'admin'`) — Scanner messages pass through.
- Debounces refresh by 250ms.
- Re-fetches the current page HTML via `fetch(window.location.href)` and replaces `#the-list`, `#cu-total-rules-count`, and `.tablenav` in place.

No CU-side changes are needed for bug 2 — the bus is already wired. This is a one-touch edit to the Scanner JS.

### Storage-event fallback

`cu-bus.js` falls back to a `localStorage` write/remove cycle when `BroadcastChannel` is unavailable. To keep parity, Scanner's emit can use the same fallback:

```js
try {
    if (typeof BroadcastChannel !== 'undefined') {
        var bc = new BroadcastChannel('code-unloader');
        bc.postMessage(msg);
        bc.close();
        return;
    }
    // Storage fallback — write-then-remove triggers 'storage' in other tabs.
    var key = 'cu-bus:code-unloader';
    localStorage.setItem(key, JSON.stringify({ t: Date.now(), msg: msg }));
    localStorage.removeItem(key);
} catch (e) { /* noop */ }
```

This duplicates a few lines of cu-bus.js but keeps Scanner free of a hard dependency on cu-bus.js loading first. Total addition: ~10 lines.

### Where in Scanner JS

Need to locate the existing push handler in `D:\AI\CU\AI Assets Scanner\AI-Assets-Scanner\admin\js\scanner.js`. The emit goes inside the success branch of the push fetch/AJAX call, after the user-facing notice fires.

### Out of scope

- Server-side notification (WP Heartbeat, transients). Browser-only signaling is sufficient when both pages are open in the same browser; opening admin from a different machine never had this sync to begin with.
- Versioning the Scanner plugin. The change is one block in one file; no version bump unless the user wants one.

### Files

- `D:\AI\CU\AI Assets Scanner\AI-Assets-Scanner\admin\js\scanner.js` — emit block on push-success.

## Bug 3 — Show rewritten asset URL when an optimizer changes it

### Goal

Make the panel honest about which URL actually loads in the browser when a script optimizer rewrites the `src` via the WordPress `script_loader_src` / `style_loader_src` filters (Perfmatters, WP Rocket page-cache mode, hash-versioning plugins, CDN URL-rewriters). Keep the registered filename visible too, so the user can reconcile what's registered vs. what ships.

**Optimizers that rewrite via HTML output buffering instead of `script_loader_src` are NOT covered** — see the Out of scope section below.

### Behaviour change

#### Capture the rewritten URL (PHP)

Add filters in `FrontendPanel::init()` (only when `?wpcu` is present, mirroring the head-buffer pattern) at `PHP_INT_MAX` priority — after every other plugin's filter. Each filter records the post-rewrite URL in a static array keyed by `handle|type`:

```php
add_filter( 'script_loader_src', [ $this, 'capture_loader_src_js' ], PHP_INT_MAX, 2 );
add_filter( 'style_loader_src',  [ $this, 'capture_loader_src_css' ], PHP_INT_MAX, 2 );
```

```php
public function capture_loader_src_js( $src, $handle ) {
    self::$rewritten_urls[ $handle . '|js' ] = (string) $src;
    return $src;
}
public function capture_loader_src_css( $src, $handle ) {
    self::$rewritten_urls[ $handle . '|css' ] = (string) $src;
    return $src;
}
```

These filters fire when WordPress emits each `<script>` / `<link>` tag — after every other filter has run. By `wp_footer` priority 1 (where `inject_panel_html()` runs), the static array contains the final URL for every asset that actually gets emitted in the page output.

#### Inject rewritten-URL patch as a late inline script (PHP)

Critical timing detail: `script_loader_src` and `style_loader_src` fire when WordPress emits each `<script>`/`<link>` tag — head scripts during `wp_print_scripts` (inside `wp_head`), footer scripts during `wp_print_footer_scripts` (inside `wp_footer`). `inject_panel_html()` runs at `wp_footer:1`, **before** any footer scripts have been printed. At that moment, `self::$rewritten_urls` is missing every footer-script entry.

We can't post-process `assets[]` at `wp_footer:1` and have the data be complete. Instead:

1. `inject_panel_html()` at `wp_footer:1` continues to emit `CDUNLOADER_DATA` via `wp_add_inline_script(..., 'before')` — same as today, with no `rewritten_src` keys.
2. New action `add_action( 'wp_footer', [ $this, 'emit_rewritten_urls' ], PHP_INT_MAX )` runs after `wp_print_footer_scripts` (priority 20) and after every other footer hook. Use `PHP_INT_MAX` (not a literal `9999`) for symmetry with the filter priorities and to defend against any plugin scheduling at the integer cap. By this point, both head and footer scripts/styles have flowed through the loader_src filters; `self::$rewritten_urls` is complete.
3. `emit_rewritten_urls()` echoes a small inline `<script>` that mutates `window.CDUNLOADER_DATA.assets[i].rewritten_src` for any handle whose captured URL differs from the registered src:

```php
public function emit_rewritten_urls(): void {
    if ( empty( self::$rewritten_urls ) ) {
        return;
    }
    ?>
    <script id="cu-rewritten-urls">
    (function () {
        if (!window.CDUNLOADER_DATA || !Array.isArray(window.CDUNLOADER_DATA.assets)) return;
        var rewrites = <?php echo wp_json_encode( self::$rewritten_urls ); ?>;
        window.CDUNLOADER_DATA.assets.forEach(function (a) {
            var key = a.handle + '|' + a.type;
            if (rewrites[key] && rewrites[key] !== a.src) {
                a.rewritten_src = rewrites[key];
            }
        });
    })();
    </script>
    <?php
}
```

This script is parsed and executed by the browser before `DOMContentLoaded` fires (script tags execute in DOM order). `panel.js`'s closure variable `assets` is the same array reference as `window.CDUNLOADER_DATA.assets`, so the mutation is visible when `_cu.renderAssets()` runs from the `DOMContentLoaded` handler.

The `panel_data['assets']` PHP-side post-processing originally proposed is therefore dropped — the JS patch handles it.

#### Display the rewritten URL (JS)

In `panel.js` `assetRow()`, change the filename rendering:

```js
var filename = '';
if (a.src) filename = a.src.split('/').pop().split('?')[0];

var rewrittenLine = '';
if (a.rewritten_src) {
    var rewrittenName = a.rewritten_src.split('/').pop().split('?')[0];
    if (rewrittenName && rewrittenName !== filename) {
        rewrittenLine = '<div class="cu-asset-src cu-asset-src--rewritten" title="'
            + esc(a.rewritten_src)
            + '">→ ' + esc(rewrittenName) + '</div>';
    }
}
```

Inserted right after the existing `<div class="cu-asset-src">` filename line. New small CSS class `cu-asset-src--rewritten` on the existing `cu-asset-src` for de-emphasis (lighter color, monospace, small caret).

#### Update the search-input placeholder (PHP)

The new comprehensive search behavior is undiscoverable if the placeholder still reads `"Filter by handle or filename…"`. Update `FrontendPanel.php:273`:

```diff
-<input id="cu-search" type="text" class="cu-search" placeholder="Filter by handle or filename…" autocomplete="off">
+<input id="cu-search" type="text" class="cu-search" placeholder="Filter by handle, filename, source, or URL…" autocomplete="off">
```

This is the only sentinel a user has for "what does the search match?" — keeping it in sync with the JS filter is the difference between tests 3a–3d being a feature or being a buried easter egg.

### Search filter — comprehensive coverage

The existing search filter (`panel.js` lines 244-246) only checks `a.handle` and `a.src`. The full URL `a.src` is searched as a string, but a query like "animo.min" matches the URL substring even when the user is thinking in filenames. This works incidentally but breaks down as soon as `rewritten_src`, source labels, or extracted filenames enter the picture.

Extend the search to match against **every** visible identifier on a row:

```js
function matchesSearch(a, q) {
    if (!q) return true;
    var hay = [
        a.handle,
        a.src,
        a.rewritten_src,
        a.source_label,
        // Extracted filenames from src and rewritten_src — handle-vs-filename mismatches
        a.src           ? a.src.split('/').pop().split('?')[0]           : '',
        a.rewritten_src ? a.rewritten_src.split('/').pop().split('?')[0] : '',
    ].filter(Boolean).join(' ').toLowerCase();
    return hay.indexOf(q) !== -1;
}
```

Replaces the inline check at lines 244-246 in `renderAssets()`. The `.filter(Boolean)` drops empty / undefined fields. Lower-cased once per row per query (no perf concern at typical asset counts).

Concrete consequences:

- Search "eb-animation-load" → finds rows whose rewritten URL contains it, regardless of registered handle.
- Search "animo" → finds `edsanimate-animo-script` (by handle) AND any row whose registered filename `animo.min.js` contains it.
- Search "Animate It!" → finds every row in that source group via `source_label`.
- Search "wp-content/plugins/animate-it" → finds rows whose full registered src contains the path fragment.

The Inline Blocks tab is not affected (no source/handle to match — that tab has its own structure).

### Why filters and not parsing HTML output

`script_loader_src` is the WordPress-official extension point for src rewrites and is hooked by Perfmatters, WP Rocket, hash-versioning plugins, and CDN URL-rewriters — the optimizers most likely to confuse a user comparing the panel against the network tab. It does **not** cover output-buffer-based optimizers (Autoptimize and friends — see Out of scope). Parsing rendered HTML for `<script src="">` tags would catch buffer-rewritten and hardcoded scripts too, but introduces output-buffering complexity and is explicitly **out of scope** per the brainstorming choice (option B was rejected).

### Out of scope

- Hardcoded `<script src="">` tags emitted directly in theme/page-builder output, not via `wp_enqueue_script`. They have no WP handle and won't appear in the panel. Bug 3 fix doesn't change that.
- `script_loader_tag` filter rewrites that change the entire `<script>` tag (not just `src`). Most optimizers use `script_loader_src` for URL changes; the few that go via `script_loader_tag` for src would be missed. If this proves common, follow-up can parse the tag.
- **HTML-output-buffering optimizers.** Autoptimize, WP-Optimize's combine mode, FlyingPress's static-bundling mode, and similar plugins rewrite asset URLs by replacing `<script>` / `<link>` tags in the rendered HTML at `wp_loaded` / `template_redirect` time, not by hooking `script_loader_src`. Bug 3 cannot detect those rewrites with this approach. The panel will continue to show the registered URL for those rows. If buffer-based detection becomes a priority, it would be a separate feature (output-buffer scan, similar to the existing `wp_head` inline-block scan).

### Files

- `src/Frontend/FrontendPanel.php` — register `script_loader_src` / `style_loader_src` filters at `PHP_INT_MAX`, store rewritten URLs in a static map, emit the inline-script patch at `wp_footer` priority `PHP_INT_MAX` via `emit_rewritten_urls()`, AND update the search-input placeholder at line 273.
- `assets/js/panel.js` — render `rewritten_src` line in `assetRow()`; replace the inline check at lines 244-246 with the comprehensive `matchesSearch()` function.
- `assets/css/panel.css` — small new class `cu-asset-src--rewritten`.

## Risk assessment

- **Bug 1 fix.** Additive — strictly grows `assets[]`. No path that today produces a row will stop producing it. Worst case: a snapshot whose handle is no longer registered shows a row with empty src — visually similar to a "stale" rule, no functional break.
- **Bug 2 fix.** Single-file JS change in Scanner plugin. The CU side is already listening; if the message format is wrong, CU's `wireCrossTabSync` filters by `msg.type === 'cu.rule.changed'` and ignores anything else. No risk to existing pushes succeeding.
- **Bug 3 fix.** Filters at `PHP_INT_MAX` are read-only — they return `$src` unchanged. The added panel data (`rewritten_src`) is optional; missing on rows where no rewrite happened. JS handles `a.rewritten_src` being undefined.

No DB schema changes. No new REST endpoints. No new dependencies. No version bump on `CDUNLOADER_DB_VERSION`. Plugin can be hot-deployed by overwriting files.
